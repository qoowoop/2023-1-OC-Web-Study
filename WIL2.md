`git add` 

- 변경 사항을 모두 Staging Area에 넣어 추적 당하는 상태로 만드는 명령어
- 수정한 파일을 staged상태로 만들 때 사용.
- Merge할 때 충돌난 상태의 파일을 resolve상태로 만들 때 사용
- `git add`는 스테이징 영역에만 올릴뿐 Git 저장소에 실질적인 영향을 못미치기 때문. 실질적인 영향을 미치는 명령어는?`git commit`

`git commit`

- Staging Area에 있는 파일을 Local Repository로 commit.
- git commit명령을 실행하면 명령을 실행하는 시점의 버전이 commit되는 것이 아니다. 마지막으로 git add명령을 실행했을 때의 버전이 commit됨.
- 따라서 git add명령을 실행한 후에 또 파일을 수정하면?**git add 명령을 다시 실행**해서 최신 버전을 Staged상태로 만들어야 한다.

****`git push`****

- Local branch를 Remote Repository로 push하는 명령어.

****`git pull`****

- 원격 서버에서 최신 커밋들을 내려받아서 현재 로컬 브랜치와 자동으로 병합하는 명령어.
- 혼자 개발하는 프로젝트에서는 pull만 써도 상관 없겠지만 여러개발자와 협업할때는 pull의 자동병합은 문제가 될때가 많음.
- fetch랑 다르게 로컬디렉로리에 변경내용을 병합.

****`git fetch`****

- 원격저장소에서 커밋된 코드를 임시 브랜치로 다 내려받음.
- 그러나 내려받은후 현재 브랜치와 자동 병합하지 않고, 그렇기 때문에 워킹디렉터리도 변화가 없음.
- 그래서 merge 명령어를 이용해서 수동 병합해야 함.

****`git pull` 과 `git fetch` 의 차이점****

- 두 명령어 모두 원격 저장소의 내용을 가져오는 명령어 이지만 pull은 가져온 변경 내용을 로컬에 자동으로 병합하며, 이 때 로컬에서 작업하다 변경된 내용을 pull할 경우 충돌이 일어날 수 있다.
- fetch는 원격 저장소에서 가져온 내용을 로컬에 영향 미치지 않으며, 병합하기 전 확인하는 용도로 사용하는 것이 좋다.
